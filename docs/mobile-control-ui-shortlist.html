移动端控车前端参考筛选

目标是先确认视觉与结构方向，不直接改控车功能。筛选标准是移动端友好、性能优先、可预览、与 BLE/IoT/车辆控制结构相近。

当前建议

保留当前 Vite + TypeScript + 原生 DOM 架构，不引入 React/Vue 大框架。UI 做产品化重构，协议、BLE、云控链路保持边界稳定。

当前项目基线

当前页面已经是轻量移动端控车雏形，但更偏工程调试页。优势是无 UI 框架、依赖少、构建快；短板是首屏信息层级、车控反馈、按钮专业感和调试功能隔离。

Vehicle Status Card

Home Assistant 车辆状态卡片，包含电量、锁状态、胎压、车辆图片、小地图、实体分组。适合作为控车首页的信息架构参考。

技术栈：TypeScript / Web Components / Home Assistant Lovelace 借鉴点：车况总览、状态 chips、车辆视觉、地图卡片、详情折叠 风险：依赖 Home Assistant 实体体系，不能直接迁移业务链路
GitHub

这个项目的截图在 README 中，建议优先打开 GitHub 查看车辆卡片效果。

Web Bluetooth Terminal

直接面向 Web Bluetooth 的 PWA，适合参考 BLE 连接、收发缓冲、离线安装和移动端终端模式。

迁移成本：低，保留当前协议层，只吸收连接状态和 PWA 经验 不适配点：UI 是终端，不是控车 App

动态控制面板参考

Adafruit Web Bluetooth Dashboard

连接设备后根据支持的 Bluetooth services 动态生成面板。适合多车型/多固件能力差异的控车 UI。

迁移成本：中低，思路可用，服务模型不同 不适配点：需要 BLE 设备才能看完整交互
在线效果 GitHub
设备边界参考

ESP32-Web-Bluetooth

强调网页由正常 Web 基础设施托管，设备只负责 BLE 数据点和控制。适合当前“前端体验重、设备端轻”的车控模型。

迁移成本：低，主要吸收边界设计 不适配点：示例 UI 很基础，不适合作为最终视觉
在线效果 GitHub

zuix-html-pwa

响应式、touch-first、抽屉布局、懒加载和 PWA 壳层参考价值高。建议只借鉴壳层，不引入整套框架。

迁移成本：中，HTML/CSS 模式可摘取 不适配点：不是车控业务，视觉需要重新定制

次级候选

这些项目有参考价值，但不建议作为主视觉或主架构来源。原因主要是过重、偏后台、偏充电桩，或与当前原生 TS 控车项目迁移成本不匹配。

evcc

EV 充电和家庭能源管理很成熟，适合参考电量、充电、实时状态，但不覆盖锁车、灯光、坐垫等车身控制。

GitHub
EV Charger Card

充电桩卡片交互清晰，适合参考主状态、快捷动作、折叠详情三层结构，但业务偏充电器。

GitHub
Casanet

本地 IoT 设备控制、轻量 dashboard 和设备抽象值得借鉴，但视觉不够车控 App 化。

GitHub
OpenRemote

资产模型、实时属性和规则引擎强，但平台级过重，更适合抽象数据模型，不适合直接套 UI。

GitHub
vite-pwa / PWABuilder

适合补齐 manifest、service worker、离线缓存和安装体验，不是视觉参考。

vite-pwa
Adminator / Cleopatra

适合看 dashboard 组件分层、CSS token、轻 JS 页面组织，但后台气质较重，不能直接用于控车主界面。

待确认方向

确认后再输出完整执行方案。当前不建议直接套任何项目，而是组合参考：车控首页取 Vehicle Status Card，BLE/PWA 取 Web Bluetooth Terminal，动态能力取 Adafruit，轻量壳取 zuix。

A. 车控首页优先先做产品化视觉、车况总览、核心控车按钮和状态反馈。
B. 性能架构优先先做 PWA、离线缓存、连接状态机 UI、调试隔离和命令反馈。
C. 完整方案一次规划视觉、结构、性能、PWA、BLE/云控边界和分阶段落地。
选择建议：如果你要先给人看效果，选 A；如果你担心低端机、弱网、离线和 BLE 稳定性，选 B；如果目标是后续直接进入开发排期，选 C。当前项目基础很轻，推荐 C，因为它不会要求立即执行代码改造，只会先把落地范围、风险和阶段拆清楚。
